SHOPIFY GIFT FLOW HANDOFF
Project: Shopify Flow - Fraud Hold & Gift Automation Fix
Reference ticket: #43
Prepared on: 2026-03-13

1. PROJECT GOAL

The business problem is that gift workflows were running on Order created, while fraud review and hold logic ran slightly later. That created a race where a free gift could be added and fulfilled even though the rest of the order was later held for fraud/review.

The quick-fix requirement from the project description was:
- make gift logic run after fraud analysis
- only proceed if the order is not high-risk
- only proceed if the order is not on hold / not under review

2. HIGH-LEVEL DESIGN THAT WAS IMPLEMENTED

The original gift flows were converted into eligibility-only flows:
- they still run on Order created
- they no longer add the gift directly
- they only add eligibility tags

A new post-fraud flow was added:
- Add Gift After Risk Review
- trigger: Order risk analyzed
- this is intended to be the only flow that actually adds the gift

Additional block logic was added:
- review / hold / fraud flows now add GIFT_BLOCKED
- the post-fraud gift flow checks that GIFT_BLOCKED is absent

This was done because Shopify Flow does not make a clean "wait until the order is safe" pattern easy when multiple workflows are involved.

3. FLOWS INVOLVED

Gift eligibility flows:
- Add gift when 2+ 4oz products ordered
- Shop App - Gift Trigger - 4oz Jars
- Shop App - Gift Trigger - Bundles

Post-fraud consumer flow:
- Add Gift After Risk Review

Blocking / review / hold flows updated to add GIFT_BLOCKED:
- Prevent Try Before You Buy Abuse
- High Quantity Order Review
- Repeat High-Volume Purchase Detection
- Hold orders if customer placed more than 1 order in last 1 hour
- Shopify High-Risk Order Hold - Slack Alert

Unrelated flow intentionally left alone:
- Add tag Order Protected

4. INTENDED FINAL LOGIC

Eligibility flows:
- run on Order created
- identify qualifying orders
- add either GIFT_ELIGIBLE_4OZ or GIFT_ELIGIBLE_BUNDLE
- do not add line items

Post-fraud gift flow:
- run on Order risk analyzed
- if order risk level is not HIGH
- if order has GIFT_ELIGIBLE_4OZ or GIFT_ELIGIBLE_BUNDLE
- if order does not have GIFT_ADDED
- if order does not have GIFT_BLOCKED
- add Universal Flare Care Gift - 1oz
- add GIFT_ADDED

Gift variant used:
- Universal Flare Care Gift - 1oz
- gid://shopify/ProductVariant/46940001599706

5. WHAT WAS VERIFIED IN SHOPIFY FLOW

Verified by browser-agent inspection:
- Add gift when 2+ 4oz products ordered is eligibility-only
- Shop App - Gift Trigger - 4oz Jars is eligibility-only
- Shop App - Gift Trigger - Bundles is eligibility-only
- Add Gift After Risk Review exists and was activated
- GIFT_BLOCKED was added to the five hold/review/fraud flows listed above

This means the "happy path" is currently failing.

Do NOT assume the ticket is fully solved just because the flow architecture was changed.

7. MOST LIKELY ROOT CAUSE

The strongest hypothesis is a timing/race issue between:
- Order created eligibility tagging
- Order risk analyzed gift insertion

Likely failure mode:
- Add Gift After Risk Review fires on Order risk analyzed
- at that exact moment, GIFT_ELIGIBLE_4OZ or GIFT_ELIGIBLE_BUNDLE is not on the order yet
- the flow exits because the eligibility tag check fails
- later, the eligibility flow adds the tag
- but the post-fraud flow does not rerun, so the gift is never added

Secondary hypothesis:
- Order risk analyzed may not be firing consistently for low-risk orders the way expected

Priority 2: FIX BASED ON EVIDENCE

If the issue is "risk flow runs before eligibility tag exists", the likely fixes are:
- move the post-risk gift flow to a trigger that can see the final state reliably
- or add a second consumer path that runs when eligibility is tagged and checks that risk is already safe
- or refactor so the post-risk flow determines eligibility directly instead of relying on a previously-added tag

If the issue is "Order risk analyzed does not run for safe orders", then the current trigger choice is wrong for the happy path and must be replaced.
